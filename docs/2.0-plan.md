Status: **all items done** · Target: a 2.0 release published under a `/v2` module path (breaking changes allowed) · Upstream baseline: [SYNC.md](../SYNC.md)

The API and layout cleanups that needed the freedom to break the public surface (#1–#7 below, in dependency order) are now **all complete**; the package is ready to tag 2.0.

## 1. Drop `FormatWithBuf`

Remove the public `FormatWithBuf(number, format, *StringBuilder)` ([phonenumberutil.go](../phonenumberutil.go)); callers use `Format` instead.

Why it's safe and worth doing:

- It exists only to mirror Java's `format(PhoneNumber, PhoneNumberFormat, StringBuilder)` overload, and it's the **one and only** public symbol that exposes the internal `StringBuilder` type — removing it un-leaks that type, which is what unblocks #2.
- The allocation it's meant to save is illusory. Every non-E164 format path ends in `prefixNumberWithCountryCallingCode`, which **inserts the calling code at index 0** and forces the buffer to shift/rebuild regardless of who allocated it. So letting the caller supply (and reuse) a buffer buys nothing in the common case.
- `Format` already does the right thing — allocate a builder, delegate, return `.String()`. Switching callers from `FormatWithBuf` to `Format` is behaviour-preserving.

It's already marked `Deprecated:` in its doc comment, so the removal is pre-announced.

## 2. Move `StringBuilder` into an internal package

**Depends on #1** — `FormatWithBuf` is the only public API that exposes `*StringBuilder`, so the type can't be hidden until it's gone.

Relocate `stringbuilder.go` into an internal package (e.g. `internal/stringbuilder`) so it's invisible to importers. `StringBuilder` exists only to mirror the slice of `java.lang.StringBuilder` the port relies on — chiefly insert-at-arbitrary-index, which neither `strings.Builder` nor `bytes.Buffer` offer — and was never meant for external use. `internal/` enforces that at compile time.

Notes:

- It's used pervasively **inside** the package — `phonenumberutil.go`, `asyoutypeformatter.go`, `phonenumbermatcher.go`, and several `_test.go` files. All stay in-repo, so they can import `internal/stringbuilder` freely.
- The currently-unexported helpers that `AsYouTypeFormatter` uses (`charAt`, `setLength`, `delete`, `substring`, `lastIndexOf`) must become **exported** so callers can reach them across the new package boundary. That's not a public-API regression: an exported name inside `internal/` is still invisible to outside importers — which is exactly why an internal package is preferable to merely lowercasing the type to `stringBuilder` (the README's phrasing). Same hiding, plus it groups cleanly with the other planned `internal/` ports of upstream's `internal/` package (the regex layer).

## 3. Move metadata into a `metadata` package

Match upstream's folder layout. Java keeps a `com.google.i18n.phonenumbers.metadata` package (with `metadata/source/*` and `metadata/init/*`) for the loading/source machinery. Our equivalent is currently spread flat across the root package:

- [metadata.go](../metadata.go) — the `//go:embed` directives for every data blob
- [metadatasource.go](../metadatasource.go) — runtime loading, region maps, `metadataContainer` (its header already reads *"Port of metadata/source/\* + MetadataLoader"*)
- the decode/load helpers in [serialize.go](../serialize.go) (`loadIntArrayMap`, `decodeUnzip`, …)
- [metadataversion.go](../metadataversion.go) — the generated `MetadataVersion` const
- [metadata_util.go](../metadata_util.go) — `NumberFormat.merge`, `PhoneNumberDesc.hasPossibleLength*`

Goal: the source/loader code (and likely the `//go:embed` blobs it reads) moves into a `metadata` package mirroring `metadata/source` + `metadata/init`.

The wrinkle to resolve during implementation is **import cycles around the proto types.** Upstream's `Phonemetadata.java` — our `phonemetadata.pb.go` (`PhoneMetadata`, `NumberFormat`, `PhoneNumberDesc`, `PhoneMetadataCollection`) — sits at the *top* level in Java, not inside `metadata`. But in Go those types are referenced throughout the root package's public API: `FormatByPattern` takes `[]*NumberFormat`, the metadata accessors return `*PhoneMetadata`, `MetadataCollection()` returns `*PhoneMetadataCollection`. If the loader moves to `metadata` while the types stay in root, then `metadata` imports root *and* root imports `metadata` → cycle.

The natural Go fix is to move the generated proto types **into** the `metadata` package too and have the root package depend on it. That diverges slightly from Java's exact layout (where the proto types are top-level) but is the only acyclic option. Decide then whether to preserve root-level ergonomics with type aliases (`type NumberFormat = metadata.NumberFormat`, etc.) or accept the qualified names — same question for the public `MetadataCollection()` and `MetadataVersion`.

## 4. Move the geocoder / carrier / timezone lookups into packages

> **Status: done.** Mirrors upstream's module layout — upstream ships these as **separate Maven modules** (`carrier`, `geocoder`) plus a shared `internal/prefixmapper`. The new API:
>
> - [geocoding/geocoding.go](../geocoding/geocoding.go) → `geocoding.ForNumber` (upstream `geocoder/.../PhoneNumberOfflineGeocoder.java`)
> - [carrier/carrier.go](../carrier/carrier.go) → `carrier.ForNumber`, `carrier.SafeDisplayName`, `carrier.WithPrefixForNumber` (upstream `carrier/PhoneNumberToCarrierMapper.java`; `WithPrefixForNumber`'s prefix return is a nyaruka extension)
> - [timezone/timezone.go](../timezone/timezone.go) → `timezone.ForPrefix`, `timezone.ForNumber`, `timezone.Unknown` (upstream `geocoder/PhoneNumberToTimeZonesMapper.java`; standalone package rather than folded into `geocoding`). `UNKNOWN_TIMEZONE` moved here as `timezone.Unknown`.
> - [internal/prefixmapper/prefixmapper.go](../internal/prefixmapper/prefixmapper.go) → the shared lookup, now a `Mapper` value that owns its own mutex + per-language cache (carrier and geocoding each construct one over their own embed). Decoupled from root — it takes the E164 string, not a `*PhoneNumber`.
>
> Each new package owns its embed, so the data relocated under it (`carrier/data/`, `geocoding/data/`, `timezone/data/`) — `go:embed` can't reach a sibling dir, same reason `metadata/data/` moved in #3. `cmd/buildmetadata` writes the new paths.

The dependency direction is clean: the new packages call *into* the core (`Format`, `GetRegionCodeForNumber`, `IsMobileNumberPortableRegion`), so they import root `phonenumbers`, not the reverse — no cycle. Root now imports none of them.

**The size win was real after all — but for a different reason than first thought.** An earlier pass measured that the Go linker already dead-code-eliminates the unused prefix-data *embeds*, and concluded the split was cosmetic. That missed `golang.org/x/text`: `PhoneNumberOfflineGeocoder` uses it for the locale fallback, and its display tables have package-init side effects the linker *can't* eliminate. Because `geocoding.go` lived in root, every consumer — even one that only calls `Parse`/`Format` — linked in ~2.6 MB of x/text CLDR data. Measured before/after, a core-only binary drops from **9.33 MB to 6.74 MB**; x/text (and the geocoding prefix data) now link in only when the `geocoding` package is imported. Carrier/timezone add only their small embeds (~80 KB / ~30 KB).

The cost, as expected, is ergonomics: `phonenumbers.GetCarrierForNumber(n, "en")` becomes `carrier.ForNumber(n, "en")`, etc. Acceptable under the `/v2` breaking-change window, and it matches upstream's per-module imports.

## 5. Move the regex layer into `internal/`

Pairs with #2 (the `StringBuilder` move) — both mirror upstream's `internal/` package, and both are internal-only helpers that just need a home invisible to importers:

- `regexCache` / `readFromRegexCache` / `regexFor` (currently in [phonenumberutil.go](../phonenumberutil.go)) → `internal/regexcache`, mirroring `internal/RegexCache.java`.
- `MatchNationalNumber` and its `match` helpers (currently in [phonenumbermatcher.go](../phonenumbermatcher.go)) → `internal/regexbasedmatcher`, mirroring `internal/RegexBasedMatcher.java` + `MatcherApi.java`.

Like #2, the catch is that `regexFor` is called from many files (`phonenumberutil.go`, `phonenumbermatcher.go`, `asyoutypeformatter.go`) and `MatchNationalNumber` is already exported and used by `shortnumberinfo.go`, so they'd stay exported across the new boundary — fine, since `internal/` keeps them off the public surface regardless. Lowest priority of the set; defer unless the structural parity is wanted.

## 6. Drop `MaybeSeparateExtensionFromPhone`

`MaybeSeparateExtensionFromPhone(rawPhone)` ([phonenumberutil.go](../phonenumberutil.go)) splits a raw string into its number and extension parts (preserving original formatting) via `EXTN_PATTERN`. Same motivation as #1 — trim public surface that doesn't pull its weight:

- **No upstream counterpart.** libphonenumber has no `separateExtension`-style public method; the closest thing is the private, `@VisibleForTesting` `maybeStripExtension(*StringBuilder)` used inside parsing. So this is a nyaruka-only public addition, and the README states the project aims to be *strictly a port*. [SYNC.md](../SYNC.md) already flags it as a deliberate divergence to revisit for API purity.
- **Unused internally.** Its only references are its own helpers (`splitAtExtensionSeparator`, `removeLeadingExtensionSeparator`) and a test — parsing strips extensions through `maybeStripExtension`, not this. Unexporting it would just leave dead code, so outright removal is the clean path.

Caveat: it's exported and a genuinely handy string-in/string-out utility, so external consumers may rely on it. Treat it like `FormatWithBuf` — mark it `Deprecated:` first, and confirm nothing in our own downstream projects calls it before dropping.

## 7. Move the metadata builder into `internal/`

**Done.** `builder.go` — our port of upstream's `tools/.../BuildMetadataFromXml.java` — was build-time-only machinery (it compiles the metadata XML into the proto types) but lived in the runtime root package, exporting a whole tool's worth of API: `BuildPhoneMetadataCollection`, `BuildAlternateFormatsMetadataCollection`, `BuildCountryCodeToRegionMap`, and the five `*E` XML element types (`PhoneNumberMetadataE`, `TerritoryE`, `NumberFormatE`, `PhoneNumberDescE`, `PossibleLengthE`). It was also the only non-test root file pulling in `encoding/xml`.

Moved to `internal/metadatabuilder`. Why this was the cleanest of the v2 breaks:

- **Pure subtraction.** No runtime consumer should ever call these — the only callers are `cmd/buildmetadata` and the two metadata test helpers (`testmetadata_test.go`, `builder_test.go`), all in-module, so they import the internal package freely. Unlike #4 there's no ergonomic cost to runtime users.
- **Upstream-layout parity.** Java keeps `BuildMetadataFromXml` in a separate `tools` module, out of the runtime library entirely; `internal/metadatabuilder` is the Go equivalent.
- The proto value types live in `metadata`, so the builder now references `metadata.PhoneMetadata` etc.; locals that hold a built metadata are renamed `md` to avoid colliding with the package identifier (upstream names them `metadata`). The `*E` types travelled with it unchanged.
