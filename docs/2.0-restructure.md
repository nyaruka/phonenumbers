# 2.0 Restructure: file-split & method-order plan

Status: **proposal** · Target: a 2.0 release (no backwards-compat constraint) · Upstream baseline: libphonenumber **v9.0.32**

## Why

The library is a port of Google's [libphonenumber](https://github.com/google/libphonenumber). Upstream ships a new release every few commits, and **~90% of those releases are pure metadata refreshes** — genuine algorithm changes to the core classes are rare. Despite that, keeping in sync is currently manual and error-prone for one structural reason:

> Almost the entire port lives in a single 3,545-line `phonenumbers.go`, in a different order from upstream, with the geocoder, carrier mapper, timezone mapper, all enums, and metadata loading all mixed in.

So when upstream changes `PhoneNumberUtil.java` at some line, there is no addressable home for it in the Go tree — every sync is a grep-and-hunt.

**Guiding principle for 2.0:** *one Go file per upstream class, with functions in the same order as the upstream source.* Then "diff upstream commit → find and apply in Go" becomes mechanical, and a scheduled sync task (the end goal) can even point a human straight at the same-named file.

This document covers **only** the file split and method ordering. The metadata-injection seam, the test-suite adoption, and the new feature implementations are tracked separately; this restructure is the foundation they sit on, and it requires **no public API change** (it's pure file movement + a few exports).

## Target layout

```
phonenumbers/
├── phonenumberutil.go      ← PhoneNumberUtil.java          (core: parse/format/validate/type/region)
├── asyoutypeformatter.go   ← AsYouTypeFormatter.java       (NEW — currently absent)
├── phonenumbermatcher.go   ← PhoneNumberMatcher.java       (NEW — currently a nil stub in matcher.go)
├── phonenumbermatch.go     ← PhoneNumberMatch.java         (NEW)
├── shortnumberinfo.go      ← ShortNumberInfo.java          (rename of shortnumber_info.go)
├── geocoding.go            ← geocoder/.../PhoneNumberOfflineGeocoder.java
├── carrier.go              ← carrier/.../PhoneNumberToCarrierMapper.java
├── timezone.go             ← geocoder/.../PhoneNumberToTimeZonesMapper.java
├── prefixmapper.go         ← internal/prefixmapper/*       (shared carrier/geocoding/timezone prefix lookup)
├── enums.go                ← the public enums (Format/Type/MatchType/ValidationResult/Leniency)
├── errors.go               ← NumberParseException.ErrorType + the package's error values
├── metadatasource.go       ← metadata/source/* + MetadataLoader  (loading & region→metadata maps)
├── phonemetadata.pb.go     ← Phonemetadata.java   (generated — unchanged)
├── phonenumber.pb.go       ← Phonenumber.java     (generated — unchanged)
├── builder.go              ← tools: BuildMetadataFromXml.java (build-time only — unchanged)
├── insertablebuffer.go     ← (Go-specific StringBuilder helper — unchanged)
├── serialize.go            ← (Go-specific binary prefix-map codec — unchanged)
└── internal/
    ├── regexcache.go       ← internal/RegexCache.java
    └── regexbasedmatcher.go← internal/RegexBasedMatcher.java + MatcherApi.java
```

`phonenumberutil.go` will still be large (~3,000 lines) — that's fine and intended, because `PhoneNumberUtil.java` is itself one ~3,700-line class. The win is that it's the *same* file with the *same* order, not that it's small.

The package stays flat as `phonenumbers` (single import is a feature, and embed+lazy-load means there's no jar-size reason to split geocoder/carrier into separate Go modules the way upstream does). The one exception is `internal/`, which mirrors upstream's `internal/` package — see [§ internal](#internal-subpackage-optional-second-pass).

### Per-file header convention

Every mirrored file gets a header pinning its upstream source and the version last reconciled, so the sync task and humans both know exactly what a file tracks:

```go
// Port of PhoneNumberUtil.java from google/libphonenumber.
// Methods are kept in upstream source order to ease syncing.
// Last reconciled against: v9.0.32
package phonenumbers
```

## Migration steps (mechanical, low-risk)

This is pure code movement within one package, so it's safe and reviewable:

1. **Land the file skeleton + headers** — create the empty target files with their upstream-source header comments.
2. **Move, don't rewrite** — relocate each function verbatim into its target file, reordering to match the tables above. Do this in commits grouped by destination file so each diff is reviewable as "these funcs moved here, unchanged."
3. **Export the four internal-but-public methods** in the same pass: `normalizeDiallableCharsOnly`, `isNumberGeographical`, `canBeInternationallyDialled`, and add the `isNumberGeographical(type,cc)` overload. (These are genuine public API in upstream.)
4. **Delete `matcher.go`**, moving its Leniency helpers into `phonenumbermatcher.go` (the stub `NewPhoneNumberMatcher` is dropped — the real one comes with the feature work).
5. **Verify each step with `go build ./...` and `go test ./...`** — no behavior changes, so the existing suite must stay green throughout.
6. **Update `README.md`** — its "Updating Metadata" section still references `gen/*.go` outputs that no longer exist.

After this lands, the **MISSING** rows and the new files become the feature backlog, and a metadata-sync diff against upstream can be applied file-by-file against the same-named Go file.

## What this unlocks

- **Metadata refreshes (the 90% case)** touch only `data/` + the generated `.pb.go`, never the hand-ported logic — already true, but now obviously so.
- **Logic changes (the rare case)** map to a single same-named, same-ordered file, so the scheduled sync task can quote the upstream hunk and point a human at the exact Go file, using the per-file "last reconciled against vX" header to scope what's new.
- **Test adoption** gets its seam: `metadatasource.go` is the one place to add the constructor that loads upstream's synthetic test metadata, so the upstream test suite can run against frozen data and stop breaking on every data refresh.
